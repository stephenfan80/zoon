# 2026-06-29 Zoon Agent 工作台 V1 验收清单

## 验收目标

验证 Zoon 是否真的解决了这个问题：

> Agent 生成一份 Markdown 后，用户可以在同一个工作台里继续围绕这份文档协作修改，而不是把内容复制到外部工具再来回沟通。

V1 验收不以“能打开链接”作为成功标准。

V1 成功标准是：

> 用户能把 Agent 的长输出变成一份 Zoon 文档，在文档里指出问题，并让 Agent 修改同一份文档。

## 标准验收场景

### 背景

用户在 Codex 里让 Agent 写一份产品方案。

示例任务：

> 帮我写一份 Zoon 作为 Agent 工作台文档协作层的 V1 产品方案，包含用户问题、目标体验、V1 范围、验收标准。

### 第一步：生成第一版 Markdown

Codex 先生成一份结构完整的 Markdown 草稿。

验收点：

- 草稿不是一句话回答，而是一份可以继续编辑的文档。
- 草稿包含标题、章节和可被局部修改的段落。

### 第二步：进入 Zoon

用户说：

> 用 Zoon 继续改。

期望行为：

- Codex 创建或打开一份 Zoon 文档。
- 文档在 Codex 工作台内可见；如果内嵌打开失败，Codex 给出完整可用链接。
- Codex 明确告诉用户：后续以这份 Zoon 文档为主版本。

验收点：

- 用户不用手动复制整篇 Markdown 到另一个工具。
- 用户能看到同一份草稿已经进入 Zoon 文档。
- Codex 没有在聊天里重新生成另一份“平行版本”。

### 第三步：文档内反馈

用户在 Zoon 文档里指出一个具体问题。

推荐验证方式：

1. 选中一段明显偏工程化的文字。
2. 留一条评论：

   > 这段太工程化，改成产品经理能读懂的用户价值表达。

验收点：

- 反馈贴在具体段落上，而不是漂在聊天框里。
- 用户不需要再复制这段文字给 Codex。

### 第四步：Agent 修改同一份文档

用户回到 Codex 说：

> 处理刚才那条评论。

期望行为：

- Codex 读取当前 Zoon 文档和评论。
- Codex 修改被评论的那一段。
- 修改结果回到同一份 Zoon 文档里。
- Codex 简短说明处理了什么。

验收点：

- 被评论段落真的改变。
- 修改后的表达更偏产品语言。
- 文档仍然是同一份 Zoon 文档。
- 用户能看出这是 Agent 修改过的内容。

### 第五步：继续第二轮

用户再提出一个局部修改：

> 把 V1 范围里的“不做什么”再收紧一点。

期望行为：

- Codex 不重新生成全文。
- Codex 只修改对应章节。
- Zoon 文档保持为主版本。

验收点：

- 第二轮修改不会丢失第一轮改动。
- 用户能继续围绕文档协作，而不是重新开始。

## 通过标准

V1 验收通过需要满足：

- Agent 的长 Markdown 输出能进入 Zoon 文档。
- 用户能在文档内提出具体反馈。
- Agent 能理解文档内反馈并修改对应位置。
- 修改结果回到同一份文档。
- 用户不用复制整篇文档或被迫离开当前 Agent 工作流。
- 失败时有清楚兜底：完整链接、当前卡点、下一步操作。

## 不通过标准

出现任一情况，V1 不算通过：

- 只是打开了 Zoon 首页或空文档，但没有承接 Agent 输出。
- 文档打开了，但后续修改仍然只发生在聊天里。
- Agent 需要用户重新复制原文才能知道要改哪里。
- Agent 创建了多份相似文档，用户不知道哪份是主版本。
- 用户评论后没有可见反馈，或者修改没有落回文档。
- Codex 声称完成，但文档内容没有变化。

## 后续扩展验收

Codex 路径跑通后，再扩展到其他工作台。

### Claude Code

- Claude Code 能把终端里的长 Markdown 输出推到 Zoon。
- 用户能在 Zoon 文档里评论。
- Claude Code 能从 Zoon 文档继续修改。

### Work Buddy

- Work Buddy 能把业务文档草稿放入 Zoon。
- 用户能在文档内审阅。
- Work Buddy 能处理文档内反馈并改回同一份文档。

## 验收口径

不要把下面这些单独当成成功：

- 插件安装成功。
- Browser 打开成功。
- 创建接口成功。
- Agent 回复“已完成”。

这些只是必要条件。

真正的成功是：

> 用户和 Agent 能围绕同一份文档持续协作。
