# Zoon 编辑页面二次优化方案

> 状态：待确认，只做需求理解和方案，不进入功能开发。
> 基线：当前 `main` 已包含上一轮编辑页优化，左侧历史文档、白底、14px 字号、右侧目录缩略、图片 URL 预览已上线。
> 本次范围：只优化 Zoon 编辑页面。历史文档排序、登录/账号绑定、文档跳转、协同协议、Agent 写入协议保持不变。

## 1. 我对这次需求的理解

这次不是重新做编辑器，而是在上一轮上线后的真实使用反馈上继续打磨。目标是让编辑页更像一个可长期工作的文档工作台，而不是“组件都在，但互相挤压”。

需要解决 5 件事：

1. 左侧历史文档列表支持收起和展开。
   - 展开时保持现在的历史文档列表。
   - 点击“收起”后，列表收到页面最左侧，只保留一个窄入口。
   - 再点击展开入口，恢复完整列表。
   - 移动端继续使用抽屉，不做桌面式拖拽收起。

2. 左侧历史文档列表支持宽度调整。
   - 桌面端可以拖动右边界调整宽度。
   - 需要设置最窄和最宽限制，不能把正文挤没，也不能窄到卡片不可读。
   - 调整后的宽度建议保存在本机浏览器，下次打开同一设备继续沿用。

3. 恢复“人类 + Agent + 修改”的颜色识别。
   - 当前代码里颜色体系仍存在，但视觉上太弱，用户感知不到。
   - 需要让“谁写的、谁改的、哪些是建议/修改”重新变得可见。
   - 不改协同数据结构，先通过样式、色条、标记可见性来恢复识别。

4. 修复评论/建议输入框唤起后的布局问题。
   - 点击“评论”或“建议”时，输入框不能再变成截图里那种横跨底部的大黑块。
   - 输入框应当尊重当前页面布局：左侧历史栏、正文区、右侧目录都要被纳入可用空间计算。
   - 桌面端应该是紧凑的上下文输入框或右侧贴边面板；移动端才使用底部 sheet。

5. 右侧目录导航不再用单独浮层。
   - 目录应直接放在文档右侧空白区域。
   - 不要再出现“竖向缩略条 + hover 弹出目录浮层”的结构。
   - 当前阅读位置高亮、点击标题滚动到位置的能力继续保留。

## 2. 截图问题归因

### 2.1 左侧历史栏

现状：

- `src/ui/editor-document-sidebar.ts` 已经有历史列表、登录提示、账号文档、本机最近文档。
- `src/index.html` 里 `--document-sidebar-width: 272px` 是固定宽度。
- 当前只有移动端的关闭按钮和抽屉状态，没有桌面端“收起/展开”和拖拽改宽。

结论：

- 这是桌面历史栏状态能力缺失，不是账号文档逻辑问题。
- 方案应只补桌面交互，不重写文档列表数据。

### 2.2 评论/建议输入框

现状：

- 评论/建议入口来自 `src/editor/plugins/mark-selection-bar.ts`。
- 输入框渲染和定位来自 `src/editor/plugins/mark-popover.ts`。
- 当前 popover 定位主要按编辑器 DOM 和整个 viewport 计算，没有把“左侧历史栏宽度 + 右侧目录占位 + 新工作区边界”完整纳入。
- `.mark-popover-sheet` 是全宽底部 sheet 样式，适合移动端，但截图里桌面端出现了类似全宽底部占位，体验明显错误。

结论：

- 这是输入框布局策略问题，不是评论/建议数据问题。
- 修复重点是桌面端和移动端分流，以及桌面端输入框的可用区域计算。

### 2.3 右侧目录导航

现状：

- `src/ui/editor-navigation.ts` 当前仍是 `outlineToggle + outlinePanel`。
- 鼠标 hover/focus 会打开目录面板。
- `src/index.html` 的 `.editor-outline-panel` 是 fixed 浮层。

结论：

- 这与本次“目录放在文档右侧空白地方，不需要单独浮层”的目标不一致。
- 需要从交互模型上改成常驻右侧目录，而不是只调 CSS。

### 2.4 颜色体系

现状：

- `src/editor/plugins/heatmap-decorations.ts` 仍在通过 `#provenance-gutter` 渲染来源色条。
- `src/editor/plugins/marks.ts` 仍给 Agent authored、comment、suggestion 等 mark 加 class。
- `src/index.html` 里也保留了 `.mark-authored-ai`、`span[data-proof="suggestion"]`、`.mark-replace-insert-ai` 等样式。
- 但当前左侧色条只有 4px，且正文内人类/Agent/修改之间的视觉分层不够明确。

结论：

- 不是颜色体系完全丢失，而是“可见性降级”。
- 方案应恢复可感知的色彩层级，同时保持编辑页干净。

## 3. 设计原则

1. 不动数据主链路。
   - 不改文档排序。
   - 不改登录、注册、退出。
   - 不改账号文档和本机最近文档兜底。
   - 不改卡片点击当前标签页跳转 `webUrl`。

2. 桌面和移动端分开治理。
   - 桌面：左侧栏常驻，可收起，可拖宽；目录常驻右侧空白区；评论/建议输入框贴近上下文。
   - 移动端：左侧栏继续抽屉；目录和评论/建议保持移动端安全样式，不强塞桌面布局。

3. 先修工作台，再修细节审美。
   - 本次最重要的是不遮挡、不挤压、不丢失信息。
   - 色彩恢复要清楚，但不要把编辑页做成高饱和标注工具。

## 4. 推荐方案

### 4.1 左侧历史文档列表：收起/展开

推荐交互：

- 展开状态：保持当前完整历史栏。
- 顶部右侧增加“收起”按钮，位置参考用户截图。
- 收起状态：左侧只保留窄栏，宽度建议 `48px`。
- 窄栏里保留一个展开按钮和“文档”入口图标/文字，点击后恢复完整历史栏。
- 收起状态下正文、顶部工具条、左侧来源色条都应自动左移，释放更多写作空间。

推荐状态存储：

- 使用本机 `localStorage`，例如：
  - `zoon.editor.documentSidebar.collapsed`
  - 值：`true` / `false`
- 这是设备级偏好，不做账号同步。原因是不同设备屏幕宽度不同，账号同步反而容易造成错位体验。

桌面 CSS 目标：

- 新增 `--document-sidebar-collapsed-width: 48px`。
- 新增 body 状态，例如 `document-sidebar-collapsed`。
- 展开时 `--document-sidebar-width-active = --document-sidebar-width`。
- 收起时 `--document-sidebar-width-active = --document-sidebar-collapsed-width`。

移动端处理：

- `max-width: 720px` 下不启用桌面收起逻辑。
- 保持当前“文档”按钮 + 抽屉。
- 不显示拖拽手柄。

### 4.2 左侧历史文档列表：宽度调整

推荐交互：

- 在历史栏右边界增加一条很轻的拖拽手柄。
- 鼠标悬停时显示可拖拽状态。
- 拖动时正文实时跟随移动。
- 双击手柄可恢复默认宽度。

推荐宽度限制：

- 默认：`272px`，沿用当前值。
- 最窄：`236px`。
- 最宽：`420px`，同时不超过视口宽度的约 `34vw`。
- 收起宽度：`48px`，不参与拖拽宽度计算。

为什么这样定：

- `236px` 以下，搜索框、账号邮箱、卡片标题、删除/移除按钮会明显拥挤。
- `420px` 以上，历史栏会变成主要内容区，违背“文档编辑页”的主任务。
- 用 `34vw` 限制超小笔记本屏，避免正文被挤压。

推荐状态存储：

- 使用本机 `localStorage`，例如 `zoon.editor.documentSidebar.width`。
- 拖拽结束时保存。
- 如果存储值越界，启动时自动 clamp 到允许范围。

### 4.3 重新展示“人类 + Agent + 修改”的颜色体系

推荐目标：

- 用户一眼能分辨：
  - 人类写的内容。
  - Agent 写的内容。
  - 当前存在评论/建议/替换修改的位置。
- 不需要每一段都变成大面积底色，避免影响写作阅读。

推荐颜色语义：

- 人类：绿色/青绿色，延续 `--human-color: #88c2a0`。
- Agent：紫色/薰衣草色，延续 `--ai-color: #b9a5e8`。
- 修改/建议：金色或蓝色区分人类建议和 Agent 建议。
- 评论：金色弱提示。
- 需要注意/flag：玫瑰色或红色弱提示。

推荐实现：

1. 左侧来源色条增强。
   - 不是简单变粗，而是让颜色更清楚。
   - 宽度建议从当前 `4px` 调到 `6px`，但放在更合适的位置。
   - 段落之间可保留连续色条，避免用户只看到碎片。

2. 正文内标记增强。
   - Agent authored 内容使用轻微紫色底 + 细下划线。
   - 人类 authored 内容如果有显式 authored mark，可以使用极弱绿色底或只在 hover/active 时显示。
   - 建议/替换保持当前插入、删除、替换语义，但提高对比度。

3. 增加一个很轻的色彩说明入口。
   - 推荐放在左侧历史栏底部或顶部工具条的一个小圆点组。
   - 文案保持短：`人类`、`Agent`、`修改`。
   - 不做大面积说明卡片，避免干扰编辑。

边界：

- 不改变 Markdown 存储。
- 不改变 marks 数据结构。
- 不改变 Agent 写入和建议接受/拒绝逻辑。

### 4.4 评论/建议输入框：兼容编辑页布局

推荐交互：

- 桌面端：
  - 点击“评论”或“建议”后，在选中文本附近打开紧凑输入框。
  - 如果选中文本右侧有空间，优先放在正文右侧空白区。
  - 如果右侧空间被目录占用或不足，放在选区下方/上方，但宽度受控。
  - 最大宽度建议 `360px - 420px`。
  - 最大高度不超过当前可视区，内容内部滚动。
  - 不显示全屏暗色遮罩。

- 移动端：
  - 保留底部 sheet 逻辑。
  - 继续兼容键盘弹起后的 visual viewport。
  - 移动端底部 sheet 不受桌面修复影响。

推荐实现：

1. 在 `mark-popover.ts` 增加桌面布局边界计算。
   - 计算工作区左边界：左侧历史栏当前 active 宽度。
   - 计算正文区域：`#editor-container` / `.milkdown .editor`。
   - 计算右侧保留空间：右侧目录常驻宽度 + 安全间距。
   - 使用这些边界决定 popover 的 left/top，而不是只看整个 viewport。

2. 明确桌面和移动端分流。
   - `isMobileTouch()` 或视口宽度命中移动时，才使用 `.mark-popover-sheet`。
   - 桌面宽屏不允许进入全宽 bottom sheet。

3. 调整 popover 视觉。
   - 背景从纯黑改成白色或接近白色，与文档工作台一致。
   - 保留清晰边框和阴影。
   - 按钮仍保持明确主次。

4. 对“建议”输入框做高度限制。
   - 文本框可 resize，但不能撑破页面。
   - 内容多时输入框内滚动，而不是把整个页面顶开。

验收重点：

- 1440px 桌面：打开评论/建议，不出现底部整屏黑块。
- 左侧栏展开、收起、拖宽三种状态下，输入框都在可见区域。
- 右侧目录存在时，输入框不会压住目录。
- 移动端仍能正常输入，不被键盘遮挡。

### 4.5 右侧目录导航：常驻在文档右侧空白处

推荐交互：

- 桌面端标题数达到当前阈值后，右侧空白区域直接展示目录列表。
- 不再有独立浮层，不再需要 hover 才出现目录内容。
- 当前阅读位置高亮。
- 点击目录项后滚动到对应标题，并保持当前项高亮。
- 目录自身可滚动，但不要抢页面滚动。

推荐布局：

- 右侧目录宽度建议 `168px - 220px`。
- 放在正文右侧空白区域，随页面固定。
- 顶部位置与正文首屏或工具条下沿对齐。
- 目录文本单行省略，长标题 hover 时可用 title 原生提示。
- 层级缩进保留，但不要过大，避免窄列不可读。

响应式策略：

- 当可用宽度不足，例如小于 `1100px`：
  - 隐藏常驻目录，保留移动/窄屏入口。
  - 或降级为当前缩略按钮，但不在桌面大屏出现浮层。
- 移动端继续底部/抽屉式处理，不强行显示右侧目录。

推荐实现：

- `src/ui/editor-navigation.ts`
  - 移除 outline 的 hover open/close 状态。
  - `outlinePanel` 改为常驻 aside 容器。
  - `renderOutlinePanel()` 在 showOutline 时始终渲染。
  - 点击目录项后不再调用 `closeOutlinePanel()`。

- `src/index.html`
  - `.editor-outline-panel` 改为右侧常驻目录样式。
  - 移除桌面 `.editor-outline-toggle` 的主视觉占位。
  - 保留移动端必要入口。

## 5. 建议开发顺序

### 第一步：布局变量和左侧栏状态

改动：

- 增加左侧栏收起/展开状态。

- 增加最小/最大宽度限制。
- 调整 `--document-sidebar-width-active` 与正文区域联动。

- 刷新页面后宽度和收起状态按本机偏好恢复。

改动：

- 把目录从“缩略条 + 浮层”改为“右侧常驻目录”。

- 没有单独浮层。

### 第三步：评论/建议输入框兼容新布局

改动：

- 桌面 popover 重新计算可用区域。
- 桌面禁用全宽底部 sheet。
- 输入框视觉改为白色工作台风格。

验证：

- 评论/建议输入框不再出现截图里的黑色底部大块。
- 左侧栏不同宽度下不遮挡。

### 第四步：颜色体系恢复可见

改动：

- 增强 provenance 色条可见度。
- 增强 Agent authored、建议/修改的正文标记对比度。
- 增加轻量色彩说明入口。

验证：

- 用户能在正文里看出 Agent 内容。

- 左侧色条能区分主要来源。
- 页面仍保持白底文档编辑器气质。

### 第五步：回归、截图、上线准备

验证：

- 静态测试更新。
- 单元测试和构建通过。
- 桌面/移动端截图对比。
- 本地编辑页真实打开验证。

3. 左侧栏宽度和收起状态只保存在本机浏览器，不做账号同步。
4. 右侧目录在桌面大屏常驻展示；窄屏降级隐藏或移动端入口，不再使用大屏 hover 浮层。
5. 桌面评论/建议输入框改为白色紧凑面板，不再使用全屏暗色遮罩。
6. 颜色体系采用“左侧来源色条 + 正文轻标记 + 小型色彩说明”的组合，不做大面积高饱和背景。

## 9. 验收标准

桌面端：

- 历史文档列表可以收起，再次点击可以展开。
- 历史文档列表可以拖动改宽，并被限制在最窄/最宽范围内。
- 历史文档登录、退出、搜索、删除/移除、当前标签页跳转逻辑不变。
- 人类、Agent、修改/建议的颜色体系重新可见。
- 点击“评论”或“建议”后，输入框不会变成底部整屏黑块。
- 右侧目录直接展示在正文右侧空白区，没有单独浮层。

移动端：

- 左侧历史文档仍是折叠抽屉。
- 不展示桌面拖拽手柄。
- 评论/建议输入框仍兼容软键盘。
- 正文不出现横向滚动。

## 10. 我的推荐结论

建议按“布局工作台优先”的顺序推进：先做左侧栏收起/改宽和右侧目录常驻，再修评论/建议输入框，最后增强颜色体系。

原因是：左侧栏宽度、右侧目录占位、评论输入框定位三者共享同一套页面几何关系。如果先单独修输入框，后面改目录和左栏还会返工。先把工作台的边界定好，后面的交互才稳。
