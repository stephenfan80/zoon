# 2026-06-29 Zoon 作为 Agent 工作台文档协作层方案

## 一句话定位

Zoon 把 Agent 生成的 Markdown，从一段聊天输出，变成一个可以继续协作修改的文档工作区。

用户不需要把 Codex、Claude Code、Work Buddy 生成的方案、报告、PRD、文章再复制到外部文档工具里。文档可以留在 Agent 工作台里继续被阅读、标注、评论、修改和确认。

## 要解决的问题

现在 Agent 已经很会写长文档，但“写完第一稿”之后，协作会断掉。

典型路径是：

1. Agent 在聊天里输出一大段 Markdown。
2. 用户发现某一段太工程化、某个判断需要补证据、某个章节顺序要调。
3. 聊天框没有稳定的选区、批注、建议修改、版本痕迹。
4. 用户只能复制到 Notion、飞书、WPS、Google Docs 或本地文件里，再回到 Agent 对话里解释“帮我改第三段”。
5. 几轮之后，聊天里的版本、外部文档里的版本、用户心里的版本开始分叉。

所以核心问题不是“Agent 不会写 Markdown”。

核心问题是：

> Agent 生成的 Markdown 还不是一个可协作的工作对象。

## 参考类比

Cowart 的价值不是“Codex 能打开一个画布网页”，而是把 AI 生成的视觉对象变成了一个可以继续指点、标注、修改的工作区。

用户可以在画布上画箭头、圈区域、写批注，然后让 Agent 基于这些可见上下文继续改。

Zoon 应该承担同样的角色，只是对象从图片/视觉稿变成 Markdown 文档。

- Cowart：生成图片或视觉想法 -> 进入画布 -> 用户标注 -> Agent 继续改图。
- Zoon：生成 Markdown 文档 -> 进入协作文档 -> 用户选中、评论、提建议 -> Agent 继续改文档。

## 产品判断

Zoon 不应该被定位成“Agent 可以访问的外部在线文档网站”。

更准确的定位是：

> Zoon 是 Agent 工作台里的文档协作层。

这个工作台可以是 Codex，也可以是 Claude Code、腾讯 Work Buddy、Cursor 或其他 Agent 产品。Zoon 负责提供那份双方都能理解的文档对象。

## 目标体验

### Codex 场景

1. 用户让 Codex 写一份方案、报告、PRD、文章或多章节分析。
2. Codex 生成第一版 Markdown。
3. 用户说：“放到 Zoon 里继续改。”
4. Codex 在工作台内打开 Zoon 文档。
5. 用户不离开 Codex，就能阅读、编辑、选中一段、评论一段、要求重写一段。
6. Codex 能理解当前文档内容和用户标注，再把修改写回同一份文档。
7. 最终产物是一份可继续协作的 Zoon 文档，而不是一条被埋在聊天记录里的长回复。

### Claude Code 场景

1. Claude Code 在终端里生成一份长 Markdown。
2. 用户说：“推到 Zoon。”
3. Claude Code 创建或更新一份 Zoon 文档，并返回完整链接。
4. 用户在 Zoon 文档里继续审阅、评论、标注。
5. 下一轮 Claude Code 从这份文档继续工作，而不是从聊天输出重新猜上下文。

### 腾讯 Work Buddy 场景

1. Work Buddy 生成周报、产品方案、会议纪要、客户分析或需求文档。
2. Zoon 作为文档协作面板承载内容。
3. 用户在文档里直接反馈。
4. Work Buddy 根据文档里的反馈继续修改对应章节。

对企业场景来说，Zoon 的价值是把 Agent 输出变成一个可审阅、可沉淀、可继续协作的业务文档。

## Zoon 必须提供的能力

### 1. 稳定的文档对象

Agent 生成的内容必须变成一份有标题、可打开、可分享、可继续修改的文档。

它不能只是聊天历史里的一段长文本。

### 2. 文档内的用户意图

用户应该能在文档里表达意图：

- 选中一段，让 Agent 改得更产品化。
- 评论一句话，说这里需要补证据。
- 标注一个章节，说结构要重排。
- 要求 Agent 扩写某一节。
- 接受或拒绝一段建议修改。

重点是：用户的意图要贴在具体文档位置上，而不是漂在聊天框里。

### 3. Agent 能读懂的文档上下文

Agent 需要知道：

- 当前文档正文是什么。
- 用户选中了哪里。
- 哪些评论还没处理。
- 哪些建议修改待确认。
- 哪些内容是人写的，哪些是 AI 写的。
- 当前文档是不是最新版。

这样 Agent 才能回答“我到底该改哪里”，而不是让用户把原文再复制一遍。

### 4. 用户看得见的 Agent 修改

Agent 修改文档后，用户应该能看见：

- 改了哪里。
- 是哪个 Agent 改的。
- 是直接改正文，还是给了建议。
- 哪条评论或任务被处理了。

这让用户保持控制感，同时不需要把每一次 Agent 修改都变成复杂审批流。

## V1 范围

V1 只验证一个完整闭环：

> Agent 输出 -> 进入 Zoon 文档 -> 用户在文档里反馈 -> Agent 修改同一份文档 -> 用户看到结果。

### V1 要做

- 把长 Markdown 输出转成 Zoon 文档。
- 在支持的工作台里直接打开 Zoon 文档。
- 尽量让用户不离开当前 Agent 工作流。
- 如果内嵌打开失败，提供完整可用链接兜底。
- Agent 能读取当前文档，并把修改写回同一份文档。
- 支持评论和建议作为明确的协作方式。
- 保留 AI/人类来源标记，让用户知道谁改了什么。

### V1 不做

- 不做完整 Notion / Google Docs / 飞书替代。
- 不做复杂多 Agent 编排。
- 不做重型审批流。
- 不做跨文档知识库。
- 不做复杂企业权限系统。
- 不把浏览器模拟输入作为主要写入方式。
- 不把产品做成只适配 Codex 的一次性插件。

## Codex 集成逻辑

Codex 里的 Zoon 应该像一个文档扩展，而不是一个外部网站跳转。

### 触发方式

Codex 应该在这些场景主动使用 Zoon：

- 用户说“推到 Zoon”。
- 用户说“这份方案放到文档里继续改”。
- 用户粘贴了一条 Zoon 文档链接。
- 用户发送 `/zoon`，表示后续长文档都进入 Zoon。
- 用户想修改之前某个 Markdown 长输出，而不是继续在聊天里补丁式修改。

### 用户看到的流程

1. Codex 创建或打开一份 Zoon 文档。
2. Codex 在工作台内打开文档面板。
3. Codex 告诉用户“文档已准备好”。
4. 用户在文档里阅读、选中、评论或继续下指令。
5. Codex 根据当前文档内容修改对应位置。
6. Codex 告诉用户改了什么，并继续以这份 Zoon 文档作为主版本。

### 成功标准

用户应该能明确感受到：

> 这份草稿不再困在聊天框里。我可以指着文档让 Agent 改，Agent 改完还在同一份文档里。

## Claude Code 集成逻辑

Claude Code 可能没有 Codex 一样的内嵌浏览器体验，所以第一阶段重点是稳定的技能流程。

1. Claude Code 把长 Markdown 创建或更新到 Zoon 文档。
2. Claude Code 返回完整可用的 Zoon 链接。
3. 用户在 Zoon 文档里审阅和评论。
4. Claude Code 下一轮从 Zoon 文档继续读取上下文并修改。

产品承诺保持一致：

> Claude Code 的长输出不再只是终端里的一段文本，而是可以继续协作的文档。

## Work Buddy 集成逻辑

腾讯 Work Buddy 的重点场景更偏业务文档：

- 周报
- 产品方案
- 会议纪要
- 销售或客户分析
- 需求文档
- 研究备忘录

Zoon 可以作为 Work Buddy 背后的文档工作区：

1. Work Buddy 起草内容。
2. Zoon 承载可编辑文档。
3. 用户在文档内反馈。
4. Work Buddy 根据反馈重写、扩写或整理对应章节。

这让 Zoon 不只是个人创作工具，而是可嵌入企业 Agent 工作流的协作文档层。

## 当前项目已有基础

Zoon 已经有很多基础能力：

- Markdown 编辑器
- 文档创建
- 可分享文档链接
- Agent 可读取的文档状态
- Agent 可写回文档
- Agent presence
- 评论
- 建议修改
- AI/人类来源标记
- Codex 插件仓
- Codex 内打开文档的 skill

现在缺的不是底层零件，而是把这些零件包装成一个清晰的 Agent 工作台文档协作闭环。

## 关键产品风险

### 风险 1：把“打开网页”误当成产品闭环

在 Codex 里打开 Zoon URL 只是入口，不是最终价值。

真正的价值是：用户能在文档里反馈，Agent 能基于文档继续修改。

### 风险 2：用户仍然觉得自己离开了 Agent 工作流

如果 Zoon 像一个外部网站，产品价值会变弱。

V1 要尽量让它像工作台里的文档面板，而不是“跳出去用另一个工具”。

### 风险 3：技术上写入成功，但产品上没有完成任务

不能只看“写入成功”。

用户要看到目标段落真的被改了、修改来源清楚、下一轮还能继续围绕同一份文档工作。

### 风险 4：过早做成 Codex 专属

Codex 是第一个验证场景，但 Zoon 的长期价值不应该只绑定 Codex。

真正可复用的是文档对象、文档反馈、Agent 修改和来源可见。

## V1 执行计划

### 第一步：重写对外定位

把 Codex 插件和 README 里的重点，从“打开 Zoon 文档”改成：

> 让 Agent 生成的 Markdown 进入可协作修改的 Zoon 文档。

用户应该第一眼明白：Zoon 不是另一个文档网站，而是 Agent 长文档输出后的协作空间。

### 第二步：定义标准动作

定义一个清楚的动作：

> Continue in Zoon / 用 Zoon 继续改

这个动作负责把当前 Agent 长输出变成 Zoon 文档，并进入后续协作模式。

### 第三步：把 Zoon 文档设为主版本

一旦进入 Zoon，Agent 后续修改应该回到同一份文档里。

不要每次都在聊天里重新生成一个完整版本，让用户自己合并。

### 第四步：跑通评论/选区到修改的闭环

最小闭环是：

1. 用户在文档里指出一处问题。
2. Agent 读取当前文档和这处反馈。
3. Agent 修改对应段落。
4. 用户在同一份文档里看到结果。

### 第五步：用真实场景验收

用一个 PM 场景做验收：

1. 让 Codex 写一份产品方案。
2. 点击或指令“用 Zoon 继续改”。
3. 在 Zoon 文档里标注一段“太工程化”。
4. 让 Codex 改成产品语言。
5. 确认同一份 Zoon 文档里的那段文字被改好。

只要这个闭环跑通，V1 就有真实产品价值。

## 近期具体工作项

1. 改 README 和插件 README，把定位改成“Agent Markdown 输出的协作文档层”。
2. 改 Codex 插件触发文案，让用户看到“继续编辑文档”而不只是“打开链接”。
3. 保留 `zoon-open-doc`，但把它定义为“打开文档工作区”的入口，而不是产品本身。
4. 增加一个完整示例：Codex 起草方案 -> 用 Zoon 继续改 -> 用户评论 -> Codex 修改同一份文档。
5. 增加一份 V1 验收清单，验收真实用户路径，而不是只验收接口是否可用。
6. 检查现有 skill 示例，避免教 Agent 走不稳定写入路径；稳定性会直接影响用户信任。

## 决策

先用 Codex 作为第一个验证场景，但产品定位保持平台中立。

不要做成：

> Zoon for Codex

而要做成：

> Zoon for agent workbench document collaboration, starting with Codex.
